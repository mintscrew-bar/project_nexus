# 내전 플랫폼을 처음 설계하며 정한 우선순위

안녕하세요. 롤 내전 플랫폼 **NEXUS**를 개발하고 있는 하루마룬입니다.

NEXUS는 리그 오브 레전드 내전을 조금 더 편하게 열고 운영하기 위해 만들고 있는 웹 서비스입니다.

처음 이 서비스를 설계할 때 가장 먼저 생각한 것은 거창한 기능이 아니었습니다. **"내전을 열 때 사람들이 실제로 귀찮아하는 과정이 뭘까?"** 였습니다.

내전을 한 번 열려면 생각보다 해야 할 일이 많습니다.

- 참가자를 모아야 하고
- 팀을 나눠야 하고
- 각 팀의 역할을 정해야 하고
- 대진표를 만들어야 하고
- 커스텀 게임을 만들고 코드를 공유해야 하고
- 경기 결과도 기록해야 합니다

각각은 따로 보면 작은 일처럼 보이지만, 방장 입장에서는 이 모든 과정을 매번 반복해야 합니다.

그래서 NEXUS의 초기 설계 목표는 단순했습니다.

> 방장이 내전 운영에 쓰는 시간을 최대한 줄이자.

이번 글에서는 NEXUS를 처음 만들 때 어떤 순서로 기능을 설계했고, 왜 그 흐름을 우선순위로 잡았는지 정리해보려고 합니다.

---

## 1. 가장 먼저 본 것은 "방 생성 방식"이었다

NEXUS에서 가장 먼저 신경 쓴 부분은 방 생성이었습니다.

내전은 결국 방 하나에서 시작합니다. 방장이 방을 만들고, 참가자들이 들어오고, 그 방 안에서 팀 편성부터 경기 진행까지 이어집니다.

그래서 방 생성 단계에서 너무 많은 것을 나중으로 미루면 안 된다고 생각했습니다. 처음부터 방장이 정해야 하는 값들을 어느 정도 명확히 잡아야 했습니다.

예를 들면 이런 것들입니다.

- 참가 인원
- 팀 구성 방식
- 대진 방식
- 공개 / 비공개 여부
- 관전 허용 여부
- 디스코드 서버 연동 여부

여기서 중요한 건 단순히 입력 폼을 만드는 것이 아니라, 방 생성 시점에 이후 흐름을 결정하는 기준을 같이 세우는 것이었습니다.

예를 들어 팀 구성 방식을 경매로 선택하면 이후에는 경매 드래프트 흐름으로 가야 하고, 자동 밸런스를 선택하면 정원이 모두 찬 뒤 바로 팀을 자동 편성해야 합니다.

즉 방 생성은 단순한 시작 화면이 아니라, **내전 전체 흐름의 방향을 결정하는 단계**였습니다.

---

## 2. 내전 룸 안에서 끝까지 처리되게 만들고 싶었다

두 번째로 중요하게 본 것은 내전 룸 안에서의 흐름이었습니다.

개인적으로 가장 피하고 싶었던 것은 사용자가 여러 도구를 왔다 갔다 해야 하는 상황이었습니다. 예를 들어 이런 식입니다.

- 팀은 사이트에서 나눈다
- 역할은 디스코드에서 말로 정한다
- 대진표는 다른 사이트에서 만든다
- 커스텀 코드는 따로 복사해서 공유한다
- 결과는 다시 채팅으로 적는다

이렇게 되면 결국 방장은 계속 중간에서 조율해야 합니다.

그래서 NEXUS에서는 가능하면 내전 룸 안에서 다음 흐름이 자연스럽게 이어지도록 설계하고 싶었습니다.

```text
방 생성
→ 참가자 입장
→ 준비
→ 팀 편성
→ 역할 선택
→ 대진표 생성
→ 토너먼트 코드 확인
→ 경기 진행
→ 결과 기록
```

이 흐름이 하나의 방 안에서 이어져야 사용자가 편하다고 봤습니다.

특히 팀 편성과 대진표는 초기 설계에서 1순위에 가까웠습니다. 내전을 운영할 때 가장 많이 반복되고, 가장 귀찮은 부분이기 때문입니다.

---

## 3. 팀 편성 방식은 처음부터 확장 가능하게 잡았다

초기에는 경매 드래프트와 스네이크 드래프트를 먼저 생각했습니다. 경매 드래프트는 팀장이 포인트로 선수를 입찰하는 방식이고, 스네이크 드래프트는 팀장이 순서대로 선수를 선택하는 방식입니다.

하지만 서비스를 만들다 보면 분명 다른 팀 구성 방식이 필요해질 것 같았습니다. 그래서 팀 구성 방식은 방의 속성으로 분리했습니다.

```ts
type TeamMode =
  | "AUCTION"
  | "SNAKE_DRAFT"
  | "AUTO_BALANCE"
  | "MANUAL_TEAM";
```

처음부터 모든 모드를 완성한 것은 아니지만, 최소한 "팀을 어떻게 만들 것인가"는 방의 주요 설정으로 분리해두었습니다.

이렇게 해두면 나중에 자동 밸런스나 자유 팀 선택 같은 방식이 추가되더라도 전체 흐름을 완전히 갈아엎지 않아도 됩니다.

실제로 이후에 자동 밸런스와 자유 팀 선택 모드를 추가할 때도, 팀 확정 이후의 흐름은 그대로 유지할 수 있었습니다.

> 팀 구성 방식은 다르다.
> 하지만 팀이 확정된 뒤에는 역할 선택으로 간다.
> 역할 선택이 끝나면 대진표로 간다.

이 공통 흐름을 유지하는 것이 중요했습니다.

---

## 4. 역할 선택을 팀 편성과 분리한 이유

팀을 나눴다고 바로 경기를 시작할 수 있는 것은 아닙니다.

롤은 포지션이 중요한 게임입니다. 같은 팀 안에서도 누가 탑을 갈지, 누가 정글을 갈지, 누가 원딜을 갈지 정해야 합니다.

그래서 NEXUS에서는 팀 편성과 역할 선택을 분리했습니다.

- **팀 편성 단계**: "누가 어느 팀인가"를 정한다
- **역할 선택 단계**: "각 팀 안에서 누가 어떤 라인을 가는가"를 정한다

이렇게 나누면 팀 구성 방식이 달라도 역할 선택은 공통으로 처리할 수 있습니다.

경매로 팀을 만들든, 스네이크 드래프트로 만들든, 자동 밸런스로 만들든, 자유 팀 선택으로 만들든, 결국 팀이 확정되면 역할 선택 단계로 넘어갑니다.

이 구조 덕분에 새로운 팀 구성 방식을 추가할 때도 역할 선택 로직을 다시 만들 필요가 없었습니다.

---

## 5. 대진표와 토너먼트 코드까지 연결하기

팀과 역할이 정해졌다면 그다음은 경기 진행입니다. 여기서 중요한 것이 대진표입니다.

내전이 단판 한 경기로 끝나면 단순하지만, 팀 수가 늘어나면 대진표가 필요합니다. NEXUS에서는 참가 팀 수와 설정에 따라 대진표를 생성하고, 경기 흐름을 관리할 수 있도록 설계했습니다.

초기에는 대진표를 단순히 "경기 목록"으로 볼 수도 있었지만, 실제 사용자는 흐름을 보고 싶어 합니다.

- 지금 어느 팀이 어느 팀과 붙는지
- 이기면 어디로 가는지
- 더블 엘리미네이션이면 패자는 어디로 내려가는지
- 결승까지 어떤 흐름으로 이어지는지

그래서 대진표는 단순 데이터가 아니라, **내전 진행 화면의 핵심 UI**라고 생각했습니다.

또한 Riot Tournament Code를 활용할 수 있도록, 경기 단위로 코드가 연결될 수 있는 구조도 고려했습니다. 방장이 매 경기마다 코드를 따로 만들고 전달하는 수고를 줄이고 싶었기 때문입니다.

---

## 6. Discord 로그인과 봇은 그다음 우선순위였다

방 생성, 팀 편성, 역할 선택, 대진표 흐름을 먼저 잡은 뒤에는 Discord 연동을 고민했습니다.

롤 내전은 대부분 Discord를 함께 사용합니다. 참가자 모집도 Discord에서 하고, 음성 채널도 Discord를 씁니다. 그래서 로그인 방식도 Discord를 우선 고려했습니다.

Discord 로그인은 단순 인증 수단이기도 하지만, 이후 봇 기능과 연결하기 위한 기반이기도 했습니다.

- 어떤 유저가 어떤 Discord 계정인지
- 어떤 서버에 봇이 추가되어 있는지
- 어떤 방이 어떤 Discord 서버와 연결되는지
- 경기 시작 시 어떤 음성 채널을 만들고 이동시킬지

이런 흐름을 만들려면 Discord 계정과 서비스 유저를 연결하는 것이 필요했습니다.

---

## 7. 웹과 Discord 봇을 어떻게 연결할 것인가

Discord 봇은 웹 서비스와 완전히 따로 놀면 의미가 없습니다.

웹에서 방이 만들어졌다면 봇도 그 방을 알아야 하고, 팀이 확정되면 봇은 참가자를 팀 음성 채널로 이동시킬 수 있어야 합니다.

그래서 Discord 봇은 별도 서비스처럼 동작하지만, 내부적으로는 NEXUS의 방 상태와 연결되도록 설계했습니다. 예를 들어 흐름은 이런 식입니다.

```text
웹에서 방 생성
→ Discord 서버 선택
→ 봇이 내전용 음성 채널 생성
→ 참가자들이 로비 음성 채널 입장
→ 팀 편성 완료
→ 봇이 팀별 음성 채널로 이동
```

이때 중요한 것은 웹과 봇의 책임을 분리하는 것이었습니다.

- **웹**: 방 상태와 내전 진행 흐름을 관리
- **봇**: Discord 서버 안에서 채널 생성, 유저 이동, 명령어 응답을 담당

이렇게 역할을 나누지 않으면 방 생성 로직 안에 Discord API 처리가 깊게 섞이고, 유지보수가 어려워질 수 있다고 봤습니다.

---

## 8. 실시간성이 필요한 부분은 WebSocket으로 처리했다

내전 룸은 실시간성이 중요합니다.

누군가 준비를 누르면 다른 사람 화면에도 바로 보여야 하고, 팀 편성 상태가 바뀌면 로비에 있는 모든 사람에게 반영되어야 합니다. 그래서 내전 룸 기능에는 WebSocket을 사용했습니다.

실시간 처리가 필요한 대표적인 기능은 다음과 같습니다.

- 참가자 입장 / 퇴장
- 준비 상태 변경
- 팀 선택 / 팀 이동
- 경매 입찰
- 스네이크 드래프트 픽
- 역할 선택
- 대진표 갱신

HTTP 요청만으로도 구현은 가능하지만, 내전 룸처럼 여러 사용자가 동시에 같은 화면을 보고 있는 경우에는 WebSocket이 더 자연스럽다고 판단했습니다.

친구 기능도 비슷한 이유로 WebSocket을 사용했습니다. 친구의 접속 상태, 친구 요청, DM 같은 기능은 사용자가 새로고침하지 않아도 바로 반영되는 쪽이 훨씬 자연스럽기 때문입니다.

---

## 9. 처음부터 모든 것을 완벽하게 만들지는 않았다

기초 설계를 할 때 가장 조심했던 부분은 너무 일찍 복잡한 구조를 만드는 것이었습니다.

처음부터 모든 기능을 추상화하고, 모든 가능성을 대비하려고 하면 오히려 개발 속도가 느려집니다. 그래서 NEXUS는 핵심 흐름을 먼저 잡는 쪽으로 진행했습니다.

```text
방 생성
→ 참가
→ 팀 편성
→ 역할 선택
→ 대진표
→ 경기 진행
```

이 흐름을 먼저 안정적으로 만들고, 그 뒤에 Discord 봇, 실시간 통신, 친구 기능, 전적, 클랜 기능을 붙여나가는 방식이었습니다.

결과적으로 이 선택은 꽤 괜찮았다고 생각합니다. 기본 흐름이 잡혀 있으니 새로운 팀 구성 방식이나 대진표 개선을 추가할 때도 어디에 붙여야 하는지 비교적 명확했습니다.

---

## 마무리

NEXUS를 처음 설계할 때 가장 중요하게 본 것은 화려한 기능이 아니었습니다. 내전을 실제로 열고 진행하는 사람이 매번 반복하는 귀찮은 흐름을 줄이는 것이 가장 중요했습니다.

그래서 초기에 가장 먼저 신경 쓴 것은 방 생성 방식과 내전 룸 안에서의 진행 흐름이었습니다. 팀을 나누고, 역할을 정하고, 대진표를 만들고, 경기 코드와 결과까지 이어지는 흐름을 하나의 서비스 안에서 처리하는 것이 1순위였습니다.

그다음에 Discord 로그인, Discord 봇, 웹과 봇의 연동, WebSocket 기반 실시간 기능을 붙여나갔습니다.

앞으로도 NEXUS는 이 기준을 유지하려고 합니다. 기능을 추가하더라도 결국 중요한 것은 하나입니다.

> 내전을 여는 사람이 덜 귀찮아지는가?
