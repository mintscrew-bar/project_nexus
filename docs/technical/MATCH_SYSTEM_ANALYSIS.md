# 매치 및 대진표 시스템 분석 보고서 (2026-04-13)

현재 프로젝트 Nexus의 매치 생성(`MatchBracketService`) 및 승자 진출(`MatchAdvancementService`) 로직을 객관적으로 분석한 결과, 공정성과 운영 효율성 측면에서 다음과 같은 문제점들이 식별되었습니다.

---

## 1. 인원수별 매치 생성 문제점 (MatchBracketService)

### 10인 (2팀, Single Match)
*   **문제**: **단판제 강제.** 현재 로직은 무조건 한 판으로 승부를 결정함.
*   **영향**: 실력 차이가 적은 내전에서 운에 의한 승부 결정 가능성이 높음. (3판 2선승제(Bo3) 옵션 부재)
*   **사이드 불공정**: 단판 랜덤 배정으로 인해 특정 사이드(블루/레드)의 지형적 이점이 승패에 직결될 수 있음.

### 15인 (3팀, Round Robin)
*   **문제**: **연속 경기 발생.** 현재 로직(`T1 vs T2` -> `T1 vs T3` -> `T2 vs T3`)은 T1이 1, 2경기를 연속으로 치르게 함.
*   **영향**: 연속 경기를 치르는 팀의 피로도 증가 및 전략 노출로 인한 불공정성 발생.
*   **개선**: 모든 팀이 경기 후 최소 1경기의 휴식 시간을 갖도록 **순환 스케줄링(Circle Method)** 적용 필요.

### 20인/40인 (4팀/8팀, Single Elimination)
*   **문제 1: 3-4위전 부재.** 준결승 패자들에 대한 순위 결정전이 없음.
*   **문제 2: 대기 시간 불균형.** 준결승 1조와 2조의 종료 시점이 다를 경우, 먼저 이긴 팀은 과도한 대기, 늦게 이긴 팀은 휴식 없이 결승 진행.
*   **영향**: 내전의 재미 요소(꼴찌 결정전 등) 감소 및 결승전의 컨디션 불균형.

### 25인~35인 (5~7팀, Forced Round Robin)
*   **치명적 문제: 경기 수 폭주.** 5~7팀일 때 무조건 리그전(Round Robin)을 강제함.
    *   5팀: 10경기
    *   6팀: 15경기
    *   **7팀: 21경기** (현실적으로 하루 내 소화 불가)
*   **영향**: 운영이 불가능한 수준의 경기 수 발생. 
*   **개선**: 5팀 이상일 경우 **부전승(Byes)을 포함한 토너먼트** 옵션 필수 제공.

## 2. 알고리즘 및 운영상 문제점 (MatchAdvancementService)

### 리그전 동률(Tie) 처리 부재
*   **문제**: 리그전 종료 후 승수가 같을 경우(예: 세 팀 모두 1승 1패) 순위를 결정하는 로직이 없음.
*   **영향**: 우승자 선정이 불가능하거나 데이터 정합성 오류 발생 가능성.
*   **개선**: 승자승(Head-to-head) -> 득실차(Game Duration/Kills 등) -> 추첨 순의 타이브레이커 로직 필요.

### 더블 엘리미네이션(DE) 복잡성
*   **문제**: 승자조 패자가 패자조(Loser's Bracket)의 어느 위치로 가는지에 대한 '드랍(Drop)' 로직의 정교함 부족.
*   **위험**: 이미 붙었던 팀과 너무 빨리 재회하거나, 대진표가 꼬여 빈 슬롯이 생길 위험.

### 사이드 배정의 편향성
*   **문제**: 매 경기마다 단순 랜덤(`Math.random()`)을 사용하여, 다판제나 재대결 시 한 팀이 계속 같은 사이드에 배정될 수 있음.
*   **개선**: 이전 경기 사이드 기록을 참조하여 **사이드 교체(Side Swapping)**를 우선 적용하는 로직 필요.
